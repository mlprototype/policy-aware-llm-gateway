# LLM Production Gateway 要件定義書

**プロジェクト名**: LLM Production Gateway
**サブタイトル**: Policy-Aware Multi-LLM Inference Platform
**版数**: v1.0
**位置付け**: 個人開発ポートフォリオ第3弾（運用統治レイヤー）

---

## 1. 目的

### 1.1 プロジェクトの目的

LLM/Agent 呼び出しを本番運用するための **運用統治レイヤー（Gateway）** を構築する。アプリケーションと LLM プロバイダの間に配置し、コスト・可用性・セキュリティ・可観測性を一元的に統治する。

### 1.2 ポートフォリオ上の位置付け

既存2プロジェクトと組み合わせ、AI システム全体の3層構造を完成させる。

| 層 | プロジェクト | 担当領域 |
|:---|:---|:---|
| 動的制御 | Agentic Control Plane | エージェント制御、状態管理、フロー設計 |
| 静的品質保証 | Spec RAG QA | 評価基盤、SLO ゲート、Quality Contract |
| **運用統治** | **LLM Production Gateway（本プロジェクト）** | **コスト・可用性・セキュリティ・可観測性** |

### 1.3 解決する課題

- LLM 呼び出しコストの暴走と不可視性
- LLM プロバイダ障害時のサービス停止
- プロンプトインジェクション・PII 流出のリスク
- 分散したエージェント・RAG 呼び出しのトレース困難
- 監査要件（誰が・いつ・何を呼んだか）への対応不足

### 1.4 達成基準

- MVP 機能6項目が動作し、ダッシュボードで可視化できる
- 既存2プロジェクトを Gateway 経由で動かせる
- 面談で「本番運用を語れる」素材として README が完成している

## 2. 非機能要件

### 2.1 性能

| 項目 | 目標値 | 備考 |
|:---|:---|:---|
| Gateway オーバーヘッド | p95 で 50ms 以内 | LLM 呼び出し本体を除く |
| スループット | 50 RPS | ローカル検証時 |
| 起動時間 | 10秒以内 | コンテナ起動からヘルスチェック成功まで |

### 2.2 可用性

| 項目 | 要件 |
|:---|:---|
| Primary LLM 障害時 | Fallback LLM へ自動切替（Circuit Breaker） |
| Gateway 自体の冗長化 | スコープ外（個人開発のため） |
| タイムアウト | プロバイダ別に設定可能、デフォルト30秒 |

### 2.3 セキュリティ

| 項目 | 要件 |
|:---|:---|
| 認証 | API Key 認証（テナント識別） |
| PII マスキング | 入力前段で実施、復元キーを安全に管理 |
| プロンプトインジェクション検知 | ルールベース + 分類器の二段防御 |
| 監査ログ | 全リクエストを不可逆形式で記録 |
| シークレット管理 | 環境変数 + `.env` 除外、本番は Secret Manager 想定 |

### 2.4 可観測性

| 項目 | 要件 |
|:---|:---|
| 分散トレーシング | OpenTelemetry 準拠 |
| メトリクス | Prometheus 形式エクスポート |
| ログ | 構造化ログ（JSON）、相関 ID 付与 |
| ダッシュボード | Grafana でコスト・レイテンシ・エラー率を可視化 |

### 2.5 拡張性

| 項目 | 要件 |
|:---|:---|
| LLM プロバイダ追加 | Provider インターフェース実装で対応可能 |
| ポリシー追加 | Policy インターフェース実装で対応可能 |
| テナント追加 | 設定ファイル更新のみで対応可能 |

### 2.6 運用性

| 項目 | 要件 |
|:---|:---|
| 設定変更 | 再起動なしでホットリロード（ポリシー・予算） |
| ヘルスチェック | `/health` `/ready` エンドポイント提供 |
| デプロイ | Docker Compose で1コマンド起動 |

## 3. MVP 機能一覧

### 3.1 MUST（必達 - 6項目）

| # | 機能 | 概要 | 設計判断 Why |
|:---:|:---|:---|:---|
| M1 | 認証・認可・レート制御 | API Key 認証、テナント別レート制限 | リソース保護の最小単位 |
| M2 | トークン/コスト計測 | 全呼び出しのトークン計測、月次予算管理 | コスト暴走防止 |
| M3 | Primary/Fallback ルーティング | Circuit Breaker による自動切替 | 可用性担保 |
| M4 | PII マスキング | 入力前の検出・置換、復元キー管理 | コンプライアンス |
| M5 | OpenTelemetry トレーシング | 相関 ID で全層を一気通貫 | デバッグ性 |
| M6 | 監査ログ | 不可逆形式、規制対応想定 | エンタープライズ要件 |

### 3.2 SHOULD（推奨 - 余力あれば）

| # | 機能 | 概要 |
|:---:|:---|:---|
| S1 | プロンプトインジェクション検知 | ルール+分類器の二段防御 |
| S2 | テナント別ポリシーエンジン | YAML でポリシー定義、ホットリロード |
| S3 | Prompt/Model バージョニング | Spec RAG QA の Quality Contract と連結 |

### 3.3 COULD（Future Work として README に明記、実装しない）

| # | 機能 | やらない理由 |
|:---:|:---|:---|
| C1 | セマンティックキャッシュ | 埋め込み選定とヒット率調整に時間が取られる |
| C2 | A/B テスト基盤 | 統計判定まで作ると工数が膨大 |
| C3 | 段階的降格 | MVP 範囲では Circuit Breaker で十分 |
